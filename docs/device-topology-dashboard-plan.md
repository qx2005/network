# 专网概览 · 设备拓扑图模块 — 规划文档

**状态**：规划稿（不写实现代码）  
**目标页面**：专网概览（Dashboard）主内容区当前统计卡片下方的空白区域（图 2 红圈示意位置）  
**版本**：0.1

---

## 1. 背景与目标

### 1.1 背景

控制台已具备网络切片、RedCap、MEC、5G LAN/VN 等配置能力；运维需要在 **一眼可读的拓扑视图** 上感知「哪些设备/环节已纳入专网并完成配置」，并与图 1、图 3 类「侧栏组件 + 画布连线」的体验叙事对齐（从高到低：先总览再钻取）。

### 1.2 目标（可验收）

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | **状态可读** | 某类设备在平台侧配置成功后，拓扑中对应节点进入 **「已开通/已上线」** 视觉态（点亮）。 |
| G2 | **链路可读** | 指向该节点的连接（逻辑「网线」）在成功前为 **灰色**，成功后为 **绿色**；失败或异常可用 **橙色/红色**（后续迭代可选）。 |
| G3 | **可编排** | 用户可从 **侧栏（或抽屉）** 将 **模块** 拖入或置入拓扑画布，形成自定义布局（与图 3 侧栏拉出组件一致）。 |
| G4 | **与概览共存** | 模块作为 Dashboard 下方 **独立区块**，不挤压现有 KPI 卡片，保持一屏内可扫视。 |

### 1.3 非目标（本期可明确排除）

- 真实网元告警联动、秒级心跳拓扑（当前平台为演示/内存态，边界需写清）。
- 与外部硬件 IDE 画布的双向同步（除非另立集成专题）。
- 三维机房、地理 GIS（若未来需要另开需求）。

## 2. 概念模型

### 2.1 拓扑三要素

1. **节点（Node）**  
   代表一类可被配置的「设备/功能实体」在拓扑中的图标 + 文案，例如：机械臂、RedCap 终端、MEC 边缘节点、UPF 分支、切片实例、VN 组等。

2. **边（Edge）**  
   节点之间的 **逻辑连接**（可视作「网线」或「数据路径」），对应产品语言中的承载关系，例如：终端—切片、切片—DNN、VN—切片、终端—MEC 节点等（具体连法见 §4）。

3. **模块库（Palette）**  
   从侧栏「拉出」的 **可放置原型**，放置后在画布上实例化为节点；同一类模块可允许多实例或单实例（见 §5 规则）。

### 2.2 状态机（节点）

建议统一四态（与现有控制台 Tag 颜色体系统筹）：

| 状态 | 含义 | 拓扑表现（建议） |
|------|------|------------------|
| `inactive` | 未放置或未开始配置 | 灰显 / 低对比描边 |
| `placed` | 已放入拓扑，平台侧尚无成功配置 | 默认白底或浅底 + 灰色描边 |
| `active` | 对应配置已成功（下发/激活） | **高亮/点亮**（如品牌绿描边或柔和 glow） |
| `error` | 校验失败或关键依赖缺失（可选） | 警示色描边 + 悬停说明 |

**边**建议二元或三元：

- `pending`：灰色实线或虚线  
- `ok`：绿色  
- `error`：红色/橙色（可选）

## 3. 信息架构与布局（Dashboard）

### 3.1 页面结构（自上而下）

1. 现有面包屑 + 标题 + 状态条（不变）  
2. **KPI 统计卡片行**（不变）  
3. **【新增】设备拓扑图区块**  
   - **标题**：如「设备拓扑 / 配置态一览」  
   - **工具条（可选）**：重置布局、适应画布、图例说明  
   - **主体**：左侧 **模块库（可折叠）** + 右侧 **拓扑画布**  
   - **空态**：未拖拽任何模块时的引导文案 + 示意动效（可选）

### 3.2 与「红圈空白区」的关系

- 拓扑区块占据当前空白 **全宽**，高度建议 **最小 360–480px**（可随内容 `min-height` + 用户拖拽缩放二期再说），保证画布可用又不把整个首页拉得过长。

## 4. 与现有配置域的映射（数据来源逻辑）

> 不写具体接口实现，仅定义 **「何种平台事件 → 哪类节点/边变绿」**，方便后续对齐 `GET` 列表与审计。

### 4.1 建议的节点类型与触发条件（示例）

| 拓扑节点类型 | 展示名称示例 | 「点亮」(active) 的判定草案 |
|-------------|--------------|------------------------------|
| `slice` | 某切片显示名 | 列表项 `status === provisioned`（或 Agent 创建即下发后的等价态） |
| `redcap_device` | RedCap 别名/SUPI 简写 | 终端已存在且可选：已绑切片 + 省电模板应用成功（按需收紧） |
| `mec_node` | MEC 节点名 | 节点记录存在（注册成功即视为接入完成） |
| `mec_rule` | 规则名 | 规则存在且 `enabled === true`（可选再加「有命中」） |
| `vn_group` | VN 显示名 | `status === active`（与现有 FiveGLanVn 模型对齐） |
| `anchor_core` | 「专网核心 / UPF 逻辑锚点」 | 可常亮或随首条切片下发成功点亮（产品定调） |

### 4.2 边的业务含义（示例）

| 边 | 语义 | `ok` 条件草案 |
|----|------|----------------|
| RedCap → Slice | 终端签约切片 | 终端的 `sliceId` 指向已 `provisioned` 的切片 |
| VN → Slice | VN 绑定切片 | `linkedSliceId` 与已存在切片一致且切片已下发 |
| Slice → Core（锚点） | 切片接入核心策略 | 切片 `provisioned` |
| MEC Rule → VN | 规则匹配 VN | 规则 `match.vnId` 与已有 VN 一致且规则启用 |

> **注意**：以上为 **逻辑拓扑**，不要求与真实物理网线一一对应；文档中应向用户说明「逻辑连接」含义，避免误导。

### 4.3 刷新策略

- 初次进入概览：并行拉取各域列表，聚合为拓扑模型。  
- 用户从子页配置返回：依赖 **路由聚焦刷新** 或 **定时轻量轮询**（如 30s）——二选一写进实现阶段需求，避免重复请求。

## 5. 交互与编辑规则

### 5.1 从侧栏「拉出」模块

1. **Palette 项**：图标 + 中文名称 + 类型 tag（工业/接入/核心 等可选）。  
2. **放置**：拖放到画布生成节点实例；若某类型 **全局唯一**（如「机械臂场景包」），第二次拖放可 **聚焦已有节点** 或 **提示已存在**（产品决策）。  
3. **连线**：  
   - **一期**：可简化 —— 边由 **配置事实自动推导**（无手动画线），减少编辑负担。  
   - **二期**：允许手动画线，仅影响展示，不回写网元。

### 5.2 与「配置成功」的联动

- 用户在任意列表页完成 **可定义为成功** 的操作后，返回概览时应看到节点/边更新。  
- 若仅内存态：刷新页面后布局是否保留需决策：  
  - **A** `localStorage` 保存画布布局 + 类型与 id 绑定  
  - **B** 每次进入使用默认布局，仅状态从 API 来  

建议在规划稿中 **倾向 A**，否则「拖入拓扑」在刷新后丢失，体验差。

## 6. 视觉与可访问性

- **颜色**：绿/灰与现有 Ant Design token、Dashboard 状态条一致；避免仅靠颜色传达信息（节点上保留 ✓/文字标签）。  
- **对比度**：深色 sidebar + 浅色 paper 背景下，画布建议使用 **浅底细点阵** 或 **极简网格**，与参考图 1、图 3 的深色硬件 IDE **可区分**（控制台仍是运维台，不是电路仿真器全深色）。  
- **动效**：点亮可做 150–300ms 过渡；避免闪烁。

## 7. 分期实施建议

| 阶段 | 交付 | 价值 |
|------|------|------|
| **P0** | 静态拓扑骨架 + 从 API 聚合状态，**自动布局**（无拖拽） | 先验证「灰/绿」与业务判定 |
| **P1** | Palette + 拖拽放置 + `localStorage` 持久化布局 | 对齐用户「侧栏拉出」诉求 |
| **P2** | 手动画线（可选）、异常态、点击钻取子页 | 增强可运维性 |

## 8. 风险与开放问题

1. **演示数据与真实语义**：需一句产品声明 —— 拓扑为 **逻辑视图**，非真实物理连线。  
2. **多切片 / 多终端**：边交叉增多时，是否需要 **自动布线算法**（P1 以后）。  
3. **权限**：viewer 是否允许拖动编辑布局？建议 **仅 operator/admin 可编辑**，viewer 只读。  
4. **Agent 批量配置**：一次成功是否 **批量点亮** 多个节点，需在 Agent 完成回调中定义事件列表。
